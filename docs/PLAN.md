# Automation Monorepo — Structure & Operations Plan

The problem this solves: you write scripts to make your life easier, then forget
they exist, forget how to run them, forget which machine they live on, and have
no record of the work they did. This plan makes every script **discoverable,
version-controlled, self-scheduling, self-documenting, and backed by a shared
data store** — across macOS, Linux, Windows, and an always-on server.

The single idea that ties it all together: **every script is a "job" — a
self-contained folder with a `manifest.yaml`.** The manifest is the one source
of truth. Everything else (the catalog you browse, the schedules installed on
each machine, the docs) is *generated from manifests*, so it can never drift out
of date.

---

## 1. Repository layout

```
automation-monorepo/
├── README.md              # front door + quickstart
├── CATALOG.md             # AUTO-GENERATED index of every job (your discovery page)
├── machines.yaml          # registry of your computers (ids, os, roles)
├── .gitignore
├── .gitattributes         # git-lfs rules for binaries/large data
│
├── jobs/                  # every script lives here, one folder per job
│   ├── _template/         # copy this to start a new job
│   │   ├── manifest.yaml
│   │   └── README.md
│   ├── <category>/
│   │   └── <job-name>/
│   │       ├── manifest.yaml   # metadata: what, how, when, where  ← source of truth
│   │       ├── main.(py|sh|ts|go)
│   │       ├── README.md       # why it exists, gotchas
│   │       └── tests/
│   └── example/hello-report/   # a working reference job
│
├── lib/                   # shared code, reused across jobs
│   ├── python/  node/  bash/  go/
│
├── data/                  # the GLOBAL, git-synced data store (see §7)
│   ├── config/            # YAML: settings shared across jobs
│   ├── state/             # SQLite DBs + per-job state (durable outputs)
│   └── inbox/             # scratch / drop zone (git-ignored)
│
├── schedules/
│   └── generated/         # AUTO-GENERATED per-OS scheduler files
│
├── tools/                 # the meta-tooling
│   ├── auto               # the CLI: list / search / run / new / schedule / doctor
│   ├── build_catalog.py
│   └── sync_schedules.py
│
└── docs/
    ├── worklog/           # dated log of everything you (and jobs) did
    └── adr/               # architecture decision records
```

Why a **monorepo** and not one repo per script: you have many small scripts on
many machines. One repo means one `git pull` to get everything everywhere, one
place to search, one CI config, and shared libraries without package-publishing
overhead. The cost — the repo grows — is negligible for text and is handled by
git-lfs for anything binary.

## 2. The job manifest (the heart of the system)

Every job folder has a `manifest.yaml`. Nothing about a job lives only in your
head or only in a crontab — it lives here, in version control. Example:

```yaml
id: hello-report              # unique, kebab-case, stable forever
name: Daily hello report
description: >
  Writes a timestamped greeting to the data store. Reference job that
  demonstrates the manifest, logging, and the data-store conventions.
category: example
owner: sumit
language: bash                # python | bash | node | go
entrypoint: main.sh           # relative to the job folder
tags: [demo, reference]

runs_on:                      # WHERE it may run — matched against machines.yaml
  os: [macos, linux, windows] # or a subset
  machines: [any]             # or explicit ids: [laptop-mac, home-server]

schedule:                     # WHEN — one declarative spec, compiled per-OS
  cron: "0 8 * * *"           # 08:00 daily (5-field cron, always UTC unless tz set)
  timezone: Asia/Kolkata
  enabled: true

runtime:
  timeout_seconds: 300
  env: []                     # names of env vars it needs (values NOT stored here)

data:                         # what it reads/writes in the store (self-documenting)
  reads:  [config/global.yaml]
  writes: [state/hello.sqlite]

health:                       # optional: how "am I still working?" is judged
  expect_run_every: 1d
```

Because the schedule lives in the manifest, you never hand-edit a crontab again.
`auto schedule sync` reads every manifest, figures out which jobs belong on
*this* machine, and writes the native scheduler config. Move to a new laptop:
`git clone`, `auto schedule sync`, done.

## 3. Discovery — never lose a script again

Three layers, so a script is findable no matter how you look:

1. **`CATALOG.md`** — regenerated by `auto catalog` (and by CI on every push). A
   single browsable table of every job: name, what it does, language, schedule,
   which machines, last-known status. This is your "what do I even have?" page.
2. **`auto list` / `auto search <term>`** — the same data from the terminal.
   `auto search invoice` finds every job tagged or described with "invoice".
3. **Convention** — jobs are grouped by `category/` folders and every job has a
   `README.md`. Predictable paths mean you (and future-you) can guess where
   something lives.

The rule that makes this work: **a job does not exist until it has a manifest.**
`auto doctor` fails CI if any job folder is missing or has an invalid manifest,
so the catalog can never silently go stale.

## 4. Version control conventions

- **One monorepo, `main` is always runnable.** Use short-lived branches for
  risky changes; small scripts can go straight to `main`.
- **Conventional commits** scoped by job: `feat(hello-report): add ret
  logic`, `fix(backup): handle spaces in paths`, `docs(worklog): 2026-07-04`.
  The scope makes `git log -- jobs/backup/` a per-job history.
- **`.gitattributes` + git-lfs** for anything binary or large (compiled Go
  binaries, `.xlsx`, images, model files) so the repo stays fast to clone.
- **Secrets never in git.** `.env` files and `data/secrets/` are git-ignored;
  manifests reference env var *names* only. Use your OS keychain or a `.env`
  synced out-of-band (see §8).
- **Tag releases per machine rollout** (optional): `git tag deployed/home-server`
  so each machine can pin to a known-good commit and you can roll back.
- **CI (GitHub Actions or similar)** on every push: lint, run job tests,
  regenerate `CATALOG.md`, and validate all manifests. This is what keeps the
  "I forgot" problem permanently solved — the machine remembers for you.

## 5. Scheduling across every OS

You declare the schedule **once** in the manifest. `auto schedule sync` compiles
it to the right native mechanism for the machine it's run on:

| OS / role         | Mechanism                     | What sync writes                          |
|-------------------|-------------------------------|-------------------------------------------|
| Linux / server    | cron **or** systemd timers    | a managed crontab block / `*.timer` units |
| macOS             | launchd                       | `~/Library/LaunchAgents/*.plist`          |
| Windows           | Task Scheduler                | `schtasks` import XML                      |

Design points that avoid the usual scheduler pain:

- **Managed block, not blind overwrite.** On cron, sync replaces only the region
  between `# >>> automation-monorepo >>>` and `# <<<` markers, so your other
  crontab entries are untouched.
- **Machine-aware.** A job with `machines: [home-server]` is only installed on
  the server. `runs_on.os` filters by platform. Nothing gets scheduled where it
  can't run.
- **Every run goes through one wrapper** (`auto run <id>`), which: loads the
  job's env, enforces the timeout, captures stdout/stderr to a per-run log,
  writes a row to the run-history DB, and records success/failure. So logging,
  timeouts, and history are uniform across all languages for free.
- **The always-on server is the default home for anything that must not miss a
  run.** Laptops sleep; jobs tagged critical should target the server. Use the
  `runs_on.machines` field to make that explicit.

Dry-run first: `auto schedule sync --dry-run` prints exactly what it would
install before touching anything.

## 6. Documentation & the work log

Two kinds of documentation, both in-repo:

- **Per-job `README.md`** — why the job exists, non-obvious gotchas, how to run
  it by hand. Lives next to the code so it's updated in the same commit.
- **Work log (`docs/worklog/`)** — a dated Markdown journal of everything you
  do. `auto log "migrated backup job to server"` appends a timestamped entry to
  today's file (`docs/worklog/2026-07-04.md`). Job runs can append here too, so
  you get one chronological record of "what happened" — human and automated.
- **ADRs (`docs/adr/`)** — short records of *why* you chose an approach, so
  decisions aren't re-litigated. One file per decision.

The worklog answers "what did I do last week?" and the catalog answers "what do
I have?" — together they're the memory you're currently missing.

## 7. Data store — what's versioned vs. machine-local

The guiding split (see ADR 0005): **git versions the things that *define* your
automations; it does not version the data those automations *produce*.**

- **`data/config/*.yaml` — VERSIONED.** Configuration and small structured data:
  settings, lookup tables, lists — anything you'd hand-edit and want reviewed in
  a diff. This is input to jobs, belongs in git, and syncs to every machine on
  `git pull`. `global.yaml` holds cross-job settings; a job can ship its own
  `config/<job>.yaml`.
- **`data/state/*.sqlite` — MACHINE-LOCAL, NOT versioned (git-ignored).** Run
  history (`runs.sqlite`), caches, scraped rows, metrics — output that a job
  accumulates. It's specific to the machine that produced it; committing it
  causes constant binary-merge conflicts and history bloat, and it means little
  on another machine. SQLite is still the right *store* (single file, real
  queries) — it just lives locally, rebuilt from inputs, not carried in git.
- **`data/inbox/` — scratch, git-ignored.** Downloads, temp exports.

**If a specific dataset genuinely must sync across machines**, don't put the
binary DB in git. Instead:

1. **Export to a diffable format** — `JSONL`/`CSV`/`YAML` under
   `data/shared/<name>.jsonl`, committed and synced. A job rebuilds the local
   SQLite view from it. Git merges text cleanly; the DB stays derived and local.
2. **Or graduate to a hosted DB** (Postgres on the always-on server) if multiple
   machines must write concurrently in real time, or the data outgrows comfor-
   table git sizes. The manifest's `data:` block means jobs don't need
   rewriting — only the connection target changes.

Rules of thumb: is it something you *wrote* to configure a job → versioned YAML.
Is it something a job *generated* → local SQLite, git-ignored, and if it must
travel, sync a diffable export, not the `.sqlite`.

Migrations for any DB whose *schema* you want versioned live in
`data/migrations/*.sql` (schema is definition, so it's tracked even though the DB
contents aren't).

## 8. Multi-machine model

- **`machines.yaml`** registers each computer: a stable `id`, its `os`, and
  `roles` (e.g. `home-server` has role `scheduler` + `data-writer`). `auto`
  detects the current machine (by hostname) and uses this to decide what to run
  and schedule here.
- **Bootstrap a new machine** in three commands: `git clone …`, `./tools/auto
  bootstrap` (installs language runtimes it detects it needs, sets up the
  git-lfs hook), `auto schedule sync`. From zero to fully scheduled.
- **Secrets** are the one thing not in git. Options, pick one: OS keychain
  (`security`/`libsecret`/Credential Manager), or an age/sops-encrypted
  `secrets.enc.yaml` that *is* safe to commit and is decrypted at runtime. The
  encrypted-file route keeps everything in the one repo — recommended if you
  want true single-source sync.

## 9. Rollout order

1. Land this scaffold; commit; push to a private remote (GitHub/Gitea).
2. Register your machines in `machines.yaml`.
3. Migrate your **existing** scripts one at a time: `auto new` to scaffold, drop
   the script in, fill the manifest, delete the old crontab entry, `auto
   schedule sync`.
4. Turn on CI so the catalog and manifest validation run automatically.
5. Keep versioned config (`data/config/*.yaml`) in git as normal; it propagates
   on `git pull`. Leave `data/state/*.sqlite` local (git-ignored). Only add a
   diffable-export sync job if a specific dataset must travel between machines.
6. From then on: every new script starts with `auto new`. It's in the catalog,
   scheduled, documented, and backed up the moment it's born.

## 10. Cheat sheet

```
auto list                 # every job
auto search backup        # find jobs
auto run <id>             # run now (with logging, timeout, history)
auto new                  # scaffold a new job from the template
auto catalog              # regenerate CATALOG.md
auto schedule sync        # install this machine's schedules (--dry-run to preview)
auto log "what I did"     # append to today's worklog
auto doctor               # validate manifests, check health, find drift
```
